# 客户端表现层框架契约设计

> 状态：当前设计口径
> 目标：定义与具体游戏无关的表现层契约，约束“逻辑层表现”和“Unity/引擎适配层”的边界。
> 重要校准：本文只讨论 Presentation/View 契约；客户端流程编排属于并列方向，详见 `客户端流程编排框架演进设计.md` 和 `客户端流程编排阶段性复盘.md`。

---

## 1. 当前结论

表现层契约仍然有必要，但它不应该承担游戏流程编排职责。

当前包里实际存在两条并列能力：

| 能力 | 当前命名空间 | 关注点 |
|---|---|---|
| 表现契约 | `AbilityKit.Game.View.Presentation` | 快照/逻辑状态如何投影成表现批次，如何被视图适配层消费 |
| 客户端流程编排 | `AbilityKit.Game.View.Flow` | 状态进入时启用哪些 feature，如何 attach/detach/tick，如何配置和校验 |

本文只定义第一条线。不要把状态机、feature 装配、场景切换、网络连接等流程能力塞进 Presentation 契约。

---

## 2. 背景

当前不同 demo 的表现层有明显差异：

| 示例 | 驱动模型 | 可抽取内容 | 不应强抽内容 |
|---|---|---|---|
| Shooter | 网络快照、插值、手写 binder/controller | 快照投影、view batch、store、stream、sink、binder 生命周期 | 具体 controller 结构 |
| MOBA | 帧同步、Entitas/ECS、feature/system 组合 | 状态到 view model/batch 的边界、binder/sink 消费契约 | ECS 内部组件和 system 组织 |

设计目标不是统一所有表现实现，而是提供稳定边界：逻辑状态变更如何进入表现层，表现层如何暴露给 Unity adapter。

---

## 3. 逻辑层表现契约

逻辑层表现契约保持纯 C#，不依赖 UnityEngine。

核心类型：

| 类型 | 职责 |
|---|---|
| `IViewBatch` | 一帧或一次表现变更批次的最小元数据 |
| `IViewModelProjector<TSnapshot,TViewBatch>` | 把逻辑快照或状态投影为 view batch |
| `IViewModelStore<TViewBatch>` | 保存当前表现态 |
| `IViewStream<TViewBatch>` | 广播 view batch |
| `IViewSink<TViewBatch>` | 消费 view batch |
| `PresentationFacade<TSnapshot,TViewBatch>` | 组合 projector/store/stream 的门面基类 |

边界原则：

- `TSnapshot` 和 `TViewBatch` 由项目定义。
- 框架不约束 batch 内部结构。
- Projector 应尽量纯函数化，方便测试。
- Store 和 Stream 是表现层状态与事件分发边界，不负责 Unity 实例生命周期。

---

## 4. Unity/引擎适配契约

Unity 适配层只做最小约束：

| 类型 | 职责 |
|---|---|
| `IViewHandle` | 一个视图实体的生命周期句柄 |
| `IViewShellLoader` | 根据类别和模型 id 加载/释放视图外壳 |
| `IViewBinder<TViewBatch>` | 消费 view batch，驱动实际视图，并支持插值 tick/rebind/clear |
| `ViewFeature<TViewBatch>` | 把 binder 接入 feature 生命周期的基础类型 |

适配层不规定 binder 内部结构。Shooter 可以继续使用手写 controller；MOBA 可以在 binder 内部驱动 Entitas system；其他项目也可以接入自己的 UI/MVVM 框架。

---

## 5. 与 Client Flow 的关系

Client Flow 决定“某个状态启用哪些 feature”。Presentation 决定“feature 内部如何把逻辑数据变成表现数据并驱动视图”。

两者关系示例：

1. HFSM 进入 Battle.InMatch。
2. Client Flow 根据 `PhaseStateFeatureSpec` 装配 `BattleViewFeature`、`BattleHudFeature`、`BattleInputFeature`。
3. `BattleViewFeature` 内部可以使用 Presentation 契约，把同步快照投影为 view batch。
4. Unity binder 消费 view batch，更新 GameObject、UI 或 ECS view entity。

这说明 Presentation 是 feature 内部可选能力，不是流程编排的上层框架。

---

## 6. 当前风险和修正

### 6.1 `IViewBatch` 元数据可能偏强

当前 `IViewBatch` 要求 `WorldId`、`Frame`、`Sequence`。这对网络/帧同步游戏合理，但对普通 UI、单机、菜单流程可能偏重。

建议后续用第二个示例验证。如果不自然，可拆成：

- 最小 marker `IViewBatch`。
- 可选元数据接口 `IFrameViewBatch`、`ISequencedViewBatch`。

### 6.2 Presentation 尚未被 MOBA 充分验证

当前契约主要来自 Shooter 经验。MOBA 仍主要通过 Entitas/ECS 和 feature 组织表现层。

建议不要强行迁移 MOBA 全部表现层，先做一个小切面：定义 `MobaBattleViewBatch`，让一个同步或 HUD 切面通过 `IViewSink<TViewBatch>` 消费，验证是否自然。

### 6.3 不要把流程 action 塞进 Presentation

场景加载、进房、连接、重连、返回大厅等属于 Client Flow 或 AbilityKit.Flow，不属于 Presentation。Presentation 只处理表现数据管线和视图消费。

---

## 7. 当前建议

短期保持现有契约，但使用方式要更克制：

1. Shooter 继续作为 Presentation 契约的主要验证样本。
2. MOBA 只选一个小切面验证，不做全量强迁移。
3. View 契约不依赖 Flow；Flow 不依赖 View。
4. 中期拆包时，Presentation 留在 `game.view.runtime`，Client Flow 移到 `game.clientflow.runtime`。

这样可以保留表现层复用价值，同时避免把流程框架、状态机和 view binder 混成一个大框架。
