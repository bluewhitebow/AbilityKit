# 客户端流程编排阶段性复盘

> 状态：阶段性架构复盘
> 范围：`com.abilitykit.game.view.runtime` 中 Presentation / Flow / Modules / Foundation，以及 MOBA 示例当前接入方式。
> 目标：判断当前方向是否在抽象层面走偏，找出遗漏、错误风险和下一轮修正优先级。

---

## 1. 总体判断

当前方向没有在核心抽象上搞错，但有三个需要立即校准的地方：

1. 包定位已经超出 `game.view` 原始语义。
2. Flow 配置已经从“运行时绑定辅助”成长为“客户端流程编排核心”，需要更正式的配置模型和验证链路。
3. MOBA 示例接入方式仍偏手写 adapter，距离正式项目还有一层“项目流程定义”和“框架运行时装配”的边界没有完全切开。

更准确地说，现在不是“表现层框架”，而是两条并列能力：

| 能力 | 当前命名空间 | 真实职责 | 当前判断 |
|---|---|---|---|
| 表现契约 | `AbilityKit.Game.View.Presentation` | 快照/状态投影为表现批次，驱动视图消费端 | 方向正确，但还没有被 MOBA/Shooter 充分反向验证 |
| 客户端流程编排 | `AbilityKit.Game.View.Flow` | 状态进入时装配 feature、生命周期、配置校验 | 方向正确，但继续放在 game.view 会越来越别扭 |
| 模块宿主 | `AbilityKit.Game.View.Modules` | feature 内部的模块组织和依赖排序 | 有价值，但命名和归属也偏流程/运行时而不是 view |
| 基础设施 | `AbilityKit.Game.View.Foundation` | pool、subscription group 等轻量工具 | 可保留，但后续拆包时要跟随使用方下沉 |

阶段性结论：当前抽象最该防的是“把 client flow 误认为 view framework 的子能力”。它们现在能共处，是因为都服务客户端侧，但概念上不是父子关系。

---

## 2. 哪些地方是正确的

### 2.1 没有重造 transition pipeline 是正确的

用户已经指出 transition 可以用 HFSM 模块实现，这个判断很关键。现在 `PhaseStateMachineSpec<TKey,TEvent>` 只描述状态、起始状态、转移和条件 id，没有直接执行 transition，也没有包装成第二套状态机运行时。

这是正确边界：

- HFSM 负责状态层级、transition、当前状态推进。
- Client Flow 负责进入状态时如何装配能力。
- AbilityKit.Flow 适合负责 enter/exit/switch 中的异步步骤，而不是负责判定 transition。

这一点不能回退。后续最多做 HFSM adapter，不应做替代 HFSM 的状态机 builder。

### 2.2 `state lifecycle -> feature assembly` 是合理核心

`PhaseFeatureHost<TContext,TFeature>`、`PhaseFeaturePlan<TContext,TFeature>`、`PhaseStateFeatureBinding<TContext,TFeature>`、`PhaseStateFeatureRegistry<TKey,TContext,TFeature>` 这组抽象，抓住的是游戏流程中比较稳定的一层：

- 进入某个状态前是否清理旧能力。
- 进入某个状态时启用哪些 feature。
- feature 如何 attach / detach / tick。
- 状态 key 如何映射到 feature 组合。

这比抽象 MVC、ECS、Binder 内部结构更安全。项目可以继续用自己的 ECS/MVC/MVVM，只要在状态边界把能力包装成 feature。

### 2.3 纯 C# 核心方向是正确的

当前核心 Flow 类型没有直接依赖 UnityEngine、UnityHFSM 或 AbilityKit.Flow，这个边界是好的。它让这些内容可以在 `src` 测试工程中单测，也给后续 JSON、ScriptableObject、Luban 配置入口留下空间。

### 2.4 Spec/Validator 方向是正确的

从 delegate binding 继续推进到 `PhaseStateFeatureSpec`、`PhaseFeatureCatalog`、`PhaseStateFeatureValidator`、`PhaseStateMachineSpec`、`PhaseStateMachineValidator` 是正确的。

这说明框架开始从“代码里组装”走向“配置可验证”。正式项目里状态流和 feature 装配必须能提前发现：

- 状态重复。
- feature id 写错。
- 起始状态缺失。
- transition 指向未知状态。
- condition id 写错。

这条线应该继续加强。

## 3. 可能遗漏或抽象不准的地方

### 3.1 包名和概念边界已经不够准

`game.view.runtime` 现在承载了：

- Presentation contract。
- Client flow orchestration。
- Module host。
- Pool/foundation helper。

这在短期能接受，但从概念上已经不准。流程编排不是 view 的子集，它可以启停 UI、网络、资源加载、输入、场景、战斗同步、调试面板。它服务 view，但不属于 view。

风险：如果继续往 `game.view` 塞 FlowNode、HFSM adapter、ScriptableObject 配置、Editor validator，会让包名误导使用者，也会让依赖关系变难拆。

建议：中期拆成 `com.abilitykit.game.clientflow.runtime`。`game.view.runtime` 保留 Presentation / Binder / ViewShell 这些表现契约。

### 3.2 `PhaseStateFeatureSpec` 和 `PhaseStateMachineSpec` 现在是可变对象

当前 Spec 用 fluent `AddState` / `AddFeature` 构建，内部是 List。作为手写配置很舒服，但作为“配置模型”还不够正式。

风险点：

- Spec 创建后仍可继续变更，运行时如果共享实例，可能被意外修改。
- `IReadOnlyList` 只限制外部接口，不代表对象不可变。
- 未来从 JSON/Luban/SO 加载时，需要更明确的构建期和运行期分界。

建议：保留 fluent builder 作为构建器，但引入冻结后的只读模型，例如：

- `PhaseStateFeatureSpecBuilder` 负责 Add。
- `PhaseStateFeatureSpec` 变成不可变快照。
- 或者提供 `Seal()` / `Freeze()`，运行时只消费 frozen spec。

这不是马上必须改，但属于正式化前必须解决的问题。

### 3.3 condition 只用 string id 还不够完整

`PhaseStateTransitionSpec` 当前只有 `ConditionId`，MOBA adapter 里硬编码判断 `BattleRequestedConditionId` 并绑定 `_battleRequested`。

这个方向没错，因为核心不应该持有 delegate。但缺了一层正式的 condition registry / resolver。

当前风险：

- condition id 的定义和实现分散在配置和 Domain adapter。
- validator 只能检查 id 是否在列表里，不能检查运行时是否能解析。
- 多个状态机共享 condition 时，没有统一命名空间或冲突策略。

建议增加项目侧 resolver 概念，但不要放执行 delegate 到 Spec 本身：

- `IPhaseConditionResolver<TKey,TEvent>` 或轻量 `PhaseConditionCatalog`。
- Spec 只写 condition id。
- Adapter 应用到 HFSM 时从 resolver 获取对应 predicate。

这样保持纯数据，又避免在 Domain 里堆 if/else。

### 3.4 缺少状态 enter/exit action 的正式模型

现在 feature binding 有 `beforeEnter` / `afterEnter` delegate，MOBA 用它处理 `_battleSessionStarted`、首帧标记、结束后回 Lobby 等逻辑。

这说明状态生命周期不止 feature 装配，还需要 action：

- enter 前重置上下文。
- enter 后派发事件。
- exit 时清理资源。
- switch 时执行 FlowNode。

当前风险：这些 action 仍是手写 delegate，无法配置、无法验证，也容易让 `GameFlowDomain` 越来越厚。

建议下一阶段不要直接把 `beforeEnter` 扩成更多 delegate，而是设计纯数据 action 引用：

- `PhaseActionRef { string Id; PhaseActionTiming Timing; }`
- `PhaseFlowRef { string Id; PhaseActionTiming Timing; }`
- 项目 adapter 用 action catalog / flow catalog 解析执行。

核心仍保持纯数据，执行留给项目或可选集成包。

### 3.5 `IPhaseGuiFeature<TContext>` 混在纯 C# Flow 核心里有点别扭

`IPhaseGuiFeature<TContext>` 本身不依赖 UnityEngine，但语义是 GUI 调试。`PhaseFeatureHost.OnGUI` 也在核心 host 里。

风险：

- 核心流程 host 被 Unity IMGUI 生命周期污染。
- 非 Unity 项目也会看到 OnGUI 概念。
- 后续如果加更多 Unity 生命周期，很容易把纯 C# 核心拖歪。

建议：

- 短期保留，避免破坏现有 MOBA 示例。
- 中期把 GUI 扩展移动到 Unity adapter 包或扩展方法，如 `PhaseFeatureHostUnityExtensions.OnGUI(...)`。
- 核心只保留 attach/detach/tick，更多生命周期用 capability interface 或 adapter 处理。

### 3.6 ModuleHost 和 PhaseFeatureHost 有相似生命周期，需要定义层级关系

`ModuleHost<TContext,TModule>` 和 `PhaseFeatureHost<TContext,TFeature>` 都有 attach/detach/tick/rebind，一者偏 feature 容器，一者偏 feature 内模块容器。

当前问题不是重复，而是层级边界没有文档化：

- Feature 是被状态机生命周期启停的能力单元。
- Module 是 feature 内部的可排序子能力。
- ModuleHost 不应该反过来决定状态，也不应该直接参与全局状态切换。

建议在文档和命名上明确：ModuleHost 属于 feature 内部组合，不是全局流程模块系统。

### 3.7 Presentation 抽象还没有被 MOBA 证明

`PresentationFacade<TSnapshot,TViewBatch>` 和 `IViewBatch` 这条线本身来自 Shooter 分层，方向合理。但 MOBA 当前仍主要通过 Entitas / feature 组织，尚未证明它能自然接入这套 Presentation contract。

风险：

- Presentation 成为 Shooter-only 抽象。
- MOBA 的 Entitas 表现模型和 batch/store/stream 模型之间映射成本过高。
- `IViewBatch` 要求 `WorldId/Frame/Sequence`，对非网络或非帧同步游戏可能偏强。

建议：下一步不要急着把 MOBA 表现层强行改成 PresentationFacade，而是先做一层薄适配验证：

- 定义最小 `MobaViewBatch` 或 `MobaBattleViewBatch`。
- 只把 battle sync 的一个小切面投影成 batch。
- 看 Entitas system 消费 batch 是否自然。

如果不自然，应回调 `IViewBatch` 的约束，而不是硬迁移。

### 3.8 MOBA 当前流程配置仍绑定 `GameFlowDomain` nested enum

`MobaFlowConfiguration` 目前引用 `GameFlowDomain.RootState`、`GameFlowDomain.RootEvent`、`GameFlowDomain.BattleState`、`GameFlowDomain.BattleEvent`。这是本轮最小安全抽取，但正式项目设计上还不够彻底。

风险：配置类仍依赖 Domain 类型，说明“流程定义”和“流程运行器”没有完全分开。

建议下一轮把 MOBA flow ids 移到独立文件，例如：

- `MobaFlowState.cs`
- `MobaFlowEvent.cs`
- 或 `MobaRootState` / `MobaBattleState` / `MobaRootEvent` / `MobaBattleEvent`

然后 `GameFlowDomain` 和 `MobaFlowConfiguration` 同时依赖这些定义。这样配置不再反向依赖 Domain。

### 3.9 当前 MOBA 文档和实现已经不一致

`MobaFlowSpec.md` 描述了 Auth、Matchmaking、Room、PostBattle、Connectivity 并行域等更完整流程，但当前实现只有 Boot、Lobby、Battle，以及 Battle 内部若干阶段。

这不是 bug，但需要标注“目标态”和“当前态”。

风险：后续读文档的人会误以为这些状态已经实现，或误判当前抽象已经支持并行域/全局中断。

建议给文档加两层：

- Current Implementation：当前 Boot/Lobby/Battle。
- Target Flow：未来 Auth/Matchmaking/Room/Battle/PostBattle/Connectivity。

### 3.10 缺少全局中断和并行域策略

正式游戏流程里，断线、重连、被踢、账号失效、资源更新等经常是横切流程。`MobaFlowSpec.md` 已经提到 Connectivity 并行域，但框架当前只覆盖普通状态机 spec。

这不是当前抽象错误，但属于明显遗漏。

建议不要马上做复杂并行状态机封装，而是先明确策略：

- 横切状态仍由 HFSM 或外层协调器处理。
- Client flow 只提供 feature 清理和 action/flow 执行能力。
- 对“全局打断”提供清理策略和诊断，不要做新的全局事件总线。

## 4. 是否存在抽象层面已经搞错的部分

### 4.1 没有根本性错误

目前没有看到必须推倒重来的抽象。核心路线仍然合理：

- 不接管项目 MVC/ECS。
- 不替代 HFSM transition。
- 不把 Unity 依赖塞进纯 C# 主体。
- 用 feature assembly 表达状态能力组合。
- 用 Spec/Validator 把配置提前验证。

### 4.2 最接近“搞错”的地方是包边界

`game.view` 这个包名已经不能准确表达 Flow/Modules。继续扩张会造成概念债。它不是代码错误，但会变成架构错误。

推荐尽早在文档中确认未来拆包，不一定马上迁移文件。代码可以先按未来拆包方式写：

- Flow 不依赖 Presentation。
- Presentation 不依赖 Flow。
- Modules 不依赖 Unity。
- HFSM adapter 不进 core。

### 4.3 第二个风险是把 Spec 当最终配置模型

当前 Spec 更像 builder + model 混合体。用来推进示例很好，但不能直接视为长期稳定配置模型。正式配置层应强调不可变、可序列化、可校验、可诊断。

### 4.4 第三个风险是 Domain 仍承载太多 adapter 逻辑

`GameFlowDomain` 已经比之前干净，但仍在处理：

- HFSM state 注册。
- condition id 到 predicate 的映射。
- feature plan 工厂。
- session callback 到 battle event 的映射。
- state binding hook。

这说明下一步不该继续往 Domain 塞配置，而应拆：

- flow definition。
- condition resolver。
- feature catalog/plan factory。
- HFSM installer。
- runtime domain orchestration。

## 5. 下一轮推荐优先级

### P0：把 MOBA flow id 从 GameFlowDomain 中独立出来

目标：让配置不再引用 Domain nested enum。

建议新增：

- `MobaRootState`
- `MobaRootEvent`
- `MobaBattleState`
- `MobaBattleEvent`

然后 `MobaFlowConfiguration` 和 `GameFlowDomain` 都依赖这些类型。

价值：这是让 MOBA 示例接近正式项目结构最直接的一步。

### P1：为 condition/action 建 catalog/resolver

目标：消除 `AddRootTransitions` 里的硬编码 condition if/else，并为后续 enter/exit flow 做准备。

建议先做项目侧，不急着进框架 core：

- `MobaFlowConditionIds`
- `MobaFlowConditionResolver`
- `MobaFlowActionIds`
- `MobaFlowActionResolver`

框架层只保留数据引用和 validator 扩展点。

### P1：更新文档，把“已实现”和“目标态”分开

现有文档有价值，但已经落后。需要明确：

- Spec/Validator 已完成第一版。
- `MobaFlowConfiguration` 已完成第一版。
- 并行域、全局中断、FlowNode/action ref 仍是设计目标。

### P2：评估拆包，不急着移动

建议先只在文档和 namespace 设计上承认未来拆包：

- `game.clientflow.runtime`
- `game.clientflow.hfsm`
- `game.clientflow.unity`
- `game.view.runtime`

当前不急着物理迁移，避免打断正在推进的示例验证。

### P2：把 GUI 生命周期从核心 host 中抽出

短期可以保持兼容。等 Flow 包边界稳定后，把 OnGUI 相关能力挪到 Unity adapter 或扩展层。

### P3：用第二个示例验证

MOBA 是复杂示例，容易让框架按复杂场景过拟合。建议找一个 Shooter/Console 的小流程验证：

- Boot -> Connect -> InGame -> End。
- 每状态 1 到 2 个 feature。
- 只验证 Spec -> Binding -> Registry -> HFSM adapter 是否自然。

## 6. 修正文档结论

此前的设计文档“简单提出”是合理的起点，但现在应更新成更强的阶段性判断：

1. `game.view` 里 Flow 抽象不是表现层内部能力，而是客户端流程编排能力。
2. HFSM 是 transition 正解，框架不应该再造 transition 运行时。
3. Spec/Validator 是正确方向，但当前 Spec 还不是最终配置模型。
4. MOBA 示例已经开始从 demo script 走向正式项目结构，但还要继续拆 Domain、Config、Resolver、Installer。
5. 最大风险不是代码错误，而是概念边界继续模糊，最终让一个包承担 view、flow、module、hfsm、unity adapter、editor config。

---

## 7. 阶段性建议

下一阶段的关键词应该是“边界校准”，不是继续堆功能。

推荐顺序：

1. 独立 MOBA flow state/event 定义。
2. 补 condition/action resolver，减少 Domain 硬编码。
3. 更新 `MobaFlowSpec.md`，区分当前态和目标态。
4. 设计 frozen spec 或 builder/model 分离。
5. 再考虑 FlowNode 引用、Unity 配置资产、编辑器校验。

这样走，当前抽象会越来越像正式项目的流程架构，而不是从一个示例里抽出来的工具集合。
