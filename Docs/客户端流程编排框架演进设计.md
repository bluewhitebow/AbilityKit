# 客户端流程编排框架演进设计

> 状态：当前设计口径
> 最近校准：已结合 `PhaseStateFeatureSpec`、`PhaseStateMachineSpec`、`MobaFlowConfiguration` 和阶段性复盘更新。
> 目标：明确客户端流程编排的定位，避免把表现契约、HFSM transition、项目 MVC/ECS 框架混成一个万能框架。

---

## 1. 当前结论

当前方向没有根本性走偏，但早期“表现层框架”这个说法已经不够准确。现在应把能力拆成两条并列线：

| 能力 | 当前命名空间 | 真实职责 | 是否替代项目框架 |
|---|---|---|---|
| 表现契约 | `AbilityKit.Game.View.Presentation` | 快照或逻辑状态投影为表现批次，并被 Unity/其他视图适配层消费 | 否，只定义边界 |
| 客户端流程编排 | `AbilityKit.Game.View.Flow` | 状态进入时装配 feature，管理生命周期、配置和校验 | 否，只编排能力 |
| 模块宿主 | `AbilityKit.Game.View.Modules` | feature 内部的模块组织、依赖排序、attach/detach/tick | 否，只服务 feature 内部组合 |

短期可以继续放在 `com.abilitykit.game.view.runtime`，但中期应考虑拆成 `com.abilitykit.game.clientflow.runtime`。流程编排不是 view 的子集，它可以启停 UI、网络、资源加载、输入、场景、同步和调试能力。

---

## 2. 正确边界

### 2.1 HFSM 负责 transition

HFSM 仍然负责：

- 大层级状态规划，例如 Boot、Lobby、Battle、Battle.Prepare、Battle.InMatch。
- transition 条件和触发事件。
- 父子状态机组织。
- 当前状态推进和状态切换。

客户端流程编排层不设计新的 transition pipeline，也不包装出第二套状态机运行时。

### 2.2 AbilityKit.Flow 负责可等待流程

`AbilityKit.Flow` 适合承载状态 enter/exit/switch 中的动作，例如：

- 连接网关。
- 加载资源或场景。
- 等待首帧。
- 等待配置加载。
- 切换期间执行一段可中断流程。

它不负责判断状态何时迁移。迁移仍由 HFSM transition 决定。

### 2.3 Client Flow 负责 state lifecycle 到 feature assembly

Flow 命名空间下的抽象负责：

- 定义阶段上下文，例如 `PhaseContext<TEntry, TRoot>`。
- 定义阶段 feature 生命周期，例如 `IPhaseFeature<TContext>`。
- 管理 feature attach/detach/tick，例如 `PhaseFeatureHost<TContext,TFeature>`。
- 管理 feature id 到工厂的注册目录，例如 `PhaseFeaturePlan<TContext,TFeature>`。
- 描述状态进入时装配哪些 feature，例如 `PhaseStateFeatureSpec` 和 `PhaseStateFeatureBinding<TContext,TFeature>`。
- 管理状态到 binding 的映射，例如 `PhaseStateFeatureRegistry<TKey,TContext,TFeature>`。
- 描述和校验状态机拓扑，例如 `PhaseStateMachineSpec<TKey,TEvent>` 和 `PhaseStateMachineValidator<TKey,TEvent>`。

它的核心职责是 state lifecycle 到 feature assembly，不是 state transition。

### 2.4 项目框架仍由项目自己决定

项目可以继续使用自己的表现或业务框架：

- MOBA 可以用 Entitas/ECS 组织战斗表现和 view model。
- Shooter 可以用手写 binder、插值控制器、快照 view batch。
- 其他项目可以用 MVC、MVVM、UI framework 或 Unity MonoBehaviour。

Client Flow 只把这些能力包装成 feature，并在状态进入/退出时启停它们。

## 3. 当前已完成的能力

| 类型 | 角色 | 当前判断 |
|---|---|---|
| `IPhase<TContext>` | 阶段生命周期接口 | 可保留，适合作为兼容入口 |
| `IPhaseFeature<TContext>` | 阶段能力接口 | 核心接口 |
| `IPhaseGuiFeature<TContext>` | GUI 调试能力 | 短期保留，中期应迁到 Unity 扩展层 |
| `PhaseFeatureHost<TContext,TFeature>` | feature 实例生命周期容器 | 核心运行时 |
| `PhaseFeaturePlan<TContext,TFeature>` | feature id 到工厂的注册目录 | 核心运行时 |
| `PhaseStateFeatureSpec` | 状态到 feature id 的纯数据描述 | 第一版已落地，但仍是可变模型 |
| `PhaseFeatureCatalog` | feature id 目录 | 第一版已落地 |
| `PhaseStateFeatureValidator` | feature spec 校验器 | 第一版已落地 |
| `PhaseStateFeatureBinding<TContext,TFeature>` | 状态进入时的运行时装配规则 | 核心运行时 |
| `PhaseStateFeatureBindingFactory` | spec 到 binding 的转换 | 第一版已落地 |
| `PhaseStateFeatureRegistry<TKey,TContext,TFeature>` | 状态 binding 查找表 | 核心运行时 |
| `PhaseStateMachineSpec<TKey,TEvent>` | 状态机拓扑纯数据描述 | 第一版已落地 |
| `PhaseStateMachineValidator<TKey,TEvent>` | 状态机拓扑校验器 | 第一版已落地 |

这些类型仍应保持纯 C#，不依赖 UnityEngine，不依赖 UnityHFSM，不依赖 AbilityKit.Flow。HFSM、Unity、FlowNode 集成都应通过 adapter 或可选包完成。

## 4. 已过期或需要修正的早期想法

### 4.1 “game.view 是表现层框架”不再准确

现在 `game.view` 包内已经包含 client flow 能力。准确说法应是：当前包临时承载了表现契约和客户端流程编排两条并列能力。

### 4.2 不再建议新增 transition pipeline

早期曾考虑过状态转换配置，但现在明确 transition 由 HFSM 负责。框架可以描述 state machine spec，并把 spec 应用到 HFSM，但不应自己执行 transition。

### 4.3 delegate binding 不是最终配置模型

`PhaseStateFeatureBinding<TContext,TFeature>` 是运行时装配对象，不是最终配置模型。正式配置应以 spec/action ref/flow ref 这种纯数据为主，运行时 adapter 再解析成 delegate。

### 4.4 当前 Spec 还不是最终形态

`PhaseStateFeatureSpec` 和 `PhaseStateMachineSpec<TKey,TEvent>` 目前是 fluent builder + mutable model 混合体。它适合推进示例，但长期应改成 builder 与不可变 spec 分离，或引入 Freeze/Seal。

---

## 5. 当前遗漏

### 5.1 condition resolver

`PhaseStateTransitionSpec<TKey,TEvent>` 只有 `ConditionId`，但缺少正式的 condition resolver。MOBA 当前仍在 Domain adapter 中用 if/else 将 condition id 绑定到 predicate。

建议：Spec 保持纯数据，项目侧提供 resolver，HFSM adapter 应用 spec 时解析 condition。

### 5.2 action / flow ref

状态生命周期不只有 feature 装配，还会有 enter 前重置、enter 后派发事件、exit 清理、switch 异步流程。当前这些逻辑仍以 `beforeEnter` / `afterEnter` delegate 存在。

建议：引入纯数据引用，例如 `PhaseActionRef`、`PhaseFlowRef`，由项目侧 action catalog / flow catalog 解析执行。

### 5.3 全局中断和并行域策略

断线、重连、被踢、账号失效、热更新等横切流程尚未正式建模。不要急着做新状态机或事件总线，优先明确：横切状态仍由 HFSM 或外层协调器处理，Client Flow 只提供清理、action/flow 执行和诊断能力。

### 5.4 Unity 生命周期扩展

`IPhaseGuiFeature<TContext>` 和 `PhaseFeatureHost.OnGUI` 目前混在核心中。短期保留，中期迁到 Unity adapter 或扩展方法，核心只保留 attach/detach/tick。

---

## 6. 包边界建议

短期：

- 保留 `AbilityKit.Game.View.Presentation` 表现契约。
- 保留 `AbilityKit.Game.View.Flow` 流程编排契约。
- 保留 `AbilityKit.Game.View.Modules` 模块宿主。
- 文档明确它们是并列能力，不互相包含。

中期：

| 包 | 内容 |
|---|---|
| `com.abilitykit.game.clientflow.runtime` | 纯 C# 流程编排、Spec、校验、feature binding |
| `com.abilitykit.game.clientflow.hfsm` | HFSM 可选集成 |
| `com.abilitykit.game.clientflow.unity` | ScriptableObject、Unity adapter、Editor 校验 |
| `com.abilitykit.game.view.runtime` | 表现契约、view batch、binder、shell loader |

当前阶段不必立刻拆包，但代码应按未来可拆包的方式写：Flow 不依赖 Presentation，Presentation 不依赖 Flow，HFSM adapter 不进 core。

---

## 7. 下一阶段推荐

下一阶段重点是边界校准，而不是继续堆功能：

1. 把 MOBA flow state/event 从 `GameFlowDomain` nested enum 独立出来。
2. 补项目侧 condition/action resolver，减少 Domain 硬编码。
3. 更新 `MobaFlowSpec.md`，明确当前实现和目标态。
4. 设计 frozen spec 或 builder/model 分离。
5. 再考虑 FlowNode 引用、Unity 配置资产和编辑器校验。

这条路线能让当前抽象继续靠近正式项目，而不是变成示例工具集合。
