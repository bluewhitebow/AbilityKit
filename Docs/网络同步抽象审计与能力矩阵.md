# 网络同步抽象审计与能力矩阵

> 阅读对象：AbilityKit 网络同步框架维护者、示例工程维护者。
>
> 文档目标：审计当前同步抽象是否把不同层级的概念混在一起，建立更稳定的能力矩阵，避免后续在 PredictRollback / AuthoritativeInterpolation / BatchStateSync / MassBattleLodSync / HybridHeroPrediction / FastReconnect / LagCompensation 上继续堆叠错误抽象。

---

## 1. 审计结论

当前方向没有整体走歪：把通用缓冲、插值时间线、网络环境模拟、对账诊断和 DemoHarness 下沉到框架层是正确的；Shooter / Moba 作为载体只保留输入、快照样本、解码、投影和世界接入，也是正确的。

但当前抽象有一个核心风险：把多个不同层级的同步能力都塞进了一个“同步模型”枚举。`NetworkSyncModel` 里既有客户端消费策略，也有服务端发布策略、兴趣管理策略、恢复流程和服务器侧判定能力。这些不是互斥选项，而是可以组合的能力。后续如果继续按单枚举推进，会在混合同步、大规模同步、快速重连和延迟补偿上反复返工。

因此后续应从“选择一个同步模型”转向“声明一个同步能力档案”。

---

## 2. 当前抽象中混在一起的概念

| 当前枚举项 | 实际概念层级 | 是否互斥 | 问题 |
| --- | --- | --- | --- |
| `PredictRollback` | 客户端本地预测 + 权威校正消费策略 | 可与远端插值、延迟补偿组合 | 适合作为客户端播放/对账策略，不应代表整局同步方案 |
| `AuthoritativeInterpolation` | 客户端消费权威快照的播放策略 | 可与本地预测组合 | 适合作为远端实体播放策略，不应排斥本地玩家预测 |
| `BatchStateSync` | 服务端快照发布频率/聚合策略 | 可与插值、AOI、LOD 组合 | 更像 `SnapshotPolicy`，不是客户端模式 |
| `MassBattleLodSync` | 服务端兴趣管理 + LOD 筛选策略 | 可与批量同步、插值组合 | 更像 `InterestPolicy` 和 `BandwidthPolicy` |
| `HybridHeroPrediction` | 多能力组合档案 | 本身就是组合 | 不应和组成它的能力平级 |
| `FastReconnect` | 会话恢复流程 | 可与所有同步策略组合 | 更像 `RecoveryPolicy`，不是同步模式 |
| `ServerRewindLagCompensation` | 服务器侧历史回溯判定能力 | 可与预测、插值、批量同步组合 | 更像 `ServerValidationPolicy` |
| `Lockstep` | 确定性输入广播/帧同步策略 | 可作为独立架构 | 需要确定性、输入延迟、断线等待、hash 校验等前置能力 |

关键修正：`NetworkSyncModel` 可以继续保留为兼容入口，但不应该继续承担完整抽象边界。它更适合变成常用能力档案的名字，底层由多个正交 policy 组成。

## 3. 推荐能力维度

### 3.1 ClientPlaybackPolicy

描述客户端如何消费本地输入和权威状态。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `None` | 客户端不做模拟或播放，仅接收事件/状态 | 未显式建模 |
| `PredictRollback` | 本地预测，收到权威状态后覆盖/回滚/重放 | Shooter 已实现 |
| `AuthoritativeInterpolation` | 权威快照延迟播放，插值消费 | Shooter/Moba 已实现 |
| `HoldLatest` | 只保持最新权威状态，不插值 | 未显式建模 |
| `ExtrapolateThenCorrect` | 短时外推，权威到达后平滑纠正 | 未实现，当前插值播放刻意不外推 |
| `HybridLocalPredictRemoteInterpolate` | 本地实体预测，远端实体插值 | 应由组合档案表达，不应单独硬编码为唯一模式 |

### 3.2 InputPolicy

描述客户端输入如何进入权威世界。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `NoClientInput` | 纯观战、AI 或服务器驱动 | 未显式建模 |
| `ImmediateSubmit` | 客户端立即提交输入，服务端按到达/映射帧处理 | Shooter 已有 |
| `InputDelayBuffer` | 客户端延迟若干帧提交/执行，换稳定性 | 未实现 |
| `ServerRemapAcceptedFrame` | 服务端返回 accepted/current/status，客户端按结果诊断 | Shooter 协议已有雏形 |
| `DeterministicBroadcast` | lockstep 输入广播，所有客户端同帧消费 | 未实现 |

### 3.3 SnapshotPolicy

描述服务端如何发布状态。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `FullSnapshot` | 全量快照 | Shooter 已有 |
| `DeltaSnapshot` | 相对上一基线增量 | Shooter 协议有基础，导入语义仍需补齐 |
| `KeyFrameSnapshot` | 周期性关键帧，支撑 delta 恢复 | 协议有概念，策略未框架化 |
| `AuthorityOverride` | 强制覆盖客户端本地预测状态 | Shooter 已有 |
| `FixedRateStateStream` | 固定频率状态流 | AuthoritativeInterpolation 可复用 |
| `BatchSnapshot` | 多实体/多帧批量快照 | 未实现 |
| `EventStream` | 技能、命中、死亡、Buff 等事件流 | 未作为同步一等能力建模 |

### 3.4 InterestPolicy

描述服务端在大量实体中选择发什么。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `AllEntities` | 所有实体都同步 | 当前小规模 demo 默认假设 |
| `OwnerRelevant` | 同步本地玩家相关实体 | 未框架化 |
| `DistanceAoi` | 按距离/区域筛选 | 未实现 |
| `TeamOrFactionAoi` | 按队伍/阵营筛选 | 未实现 |
| `PriorityBudget` | 在带宽预算内按优先级选择 | 未实现 |
| `LodFrequency` | 不同实体使用不同同步频率 | 未实现 |

### 3.5 RecoveryPolicy

描述断线、漂移、缺包、切后台、晚进、重连如何恢复。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `None` | 不提供自动恢复 | 未显式建模 |
| `RequestFullSnapshot` | 请求定向全量快照 | Shooter 可通过重复订阅触发，专用 opcode 待补 |
| `RequestKeyFrame` | 请求关键帧恢复 delta 链 | 未实现 |
| `RequestAoiSlice` | 只请求当前兴趣区域切片 | 未实现 |
| `CatchUpToServerFrame` | 根据 server ticks / anchor 追帧 | Shooter 已有 |
| `ReconnectResume` | 携带 session/world/frame/hash 恢复 | 协议方向已有，框架能力未成型 |

### 3.6 ServerValidationPolicy

描述服务器如何处理客户端声明或请求。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `AuthoritativeOnly` | 完全以服务器状态为准 | 当前基础假设 |
| `InputValidation` | 校验输入合法性、频率和帧窗口 | 局部存在，未框架化 |
| `LagCompensatedHitValidation` | 按客户端请求帧回溯服务器历史验证命中 | 框架 helper 已实现 |
| `ClientHashAudit` | 客户端上报 hash，服务器主动审计漂移 | 文档方向有，未实现 |
| `AntiCheatEnvelope` | 速度、位置、技能冷却等反作弊边界 | 未框架化 |

### 3.7 TransportConditionPolicy

描述网络环境和传输扰动。

| 取值 | 含义 | 当前状态 |
| --- | --- | --- |
| `Ideal` | 无延迟/无丢包 | 已实现 profile |
| `Lan` | 低延迟 | 已实现 profile |
| `Mobile4G` | 中等延迟和抖动 | 已实现 profile |
| `CrossRegion` | 高延迟 | 已实现 profile |
| `PoorWifi` | 高抖动和丢包 | 已实现 profile |
| `BandwidthBudget` | 限制带宽预算并触发优先级选择 | profile 字段有基础，策略未完整打通 |

## 4. 用能力档案表达现有“同步模型”

### 4.1 PredictRollback 档案

| 维度 | 取值 |
| --- | --- |
| ClientPlaybackPolicy | `PredictRollback` |
| InputPolicy | `ImmediateSubmit` + `ServerRemapAcceptedFrame` |
| SnapshotPolicy | `FullSnapshot` + `AuthorityOverride`，后续可补 `DeltaSnapshot` |
| InterestPolicy | `AllEntities` |
| RecoveryPolicy | `CatchUpToServerFrame` + `RequestFullSnapshot` |
| ServerValidationPolicy | `AuthoritativeOnly`，可组合 `LagCompensatedHitValidation` |
| TransportConditionPolicy | 任意 profile |

### 4.2 AuthoritativeInterpolation 档案

| 维度 | 取值 |
| --- | --- |
| ClientPlaybackPolicy | `AuthoritativeInterpolation` |
| InputPolicy | 可为 `NoClientInput` 或独立本地输入通道 |
| SnapshotPolicy | `FixedRateStateStream`，可补 `KeyFrameSnapshot` |
| InterestPolicy | 当前 `AllEntities`，后续可接 AOI |
| RecoveryPolicy | `RequestKeyFrame` 或 `RequestFullSnapshot`，当前未完整建模 |
| ServerValidationPolicy | `AuthoritativeOnly` |
| TransportConditionPolicy | 任意 profile |

### 4.3 HybridHeroPrediction 档案

| 维度 | 取值 |
| --- | --- |
| ClientPlaybackPolicy | 本地英雄 `PredictRollback`，远端玩家 `AuthoritativeInterpolation`，普通单位可 `BatchSnapshot` 播放 |
| InputPolicy | 本地英雄 `ImmediateSubmit` + accepted frame 反馈 |
| SnapshotPolicy | 英雄/关键对象高频，普通单位低频 batch，关键事件走 event stream |
| InterestPolicy | `OwnerRelevant` + `DistanceAoi` + `PriorityBudget` |
| RecoveryPolicy | 按对象类型选择 full/keyframe/slice |
| ServerValidationPolicy | `InputValidation` + 可选 `LagCompensatedHitValidation` |
| TransportConditionPolicy | 任意 profile |

这说明 `HybridHeroPrediction` 不应该是与 `PredictRollback` / `AuthoritativeInterpolation` 平级的底层模式，而应该是一个组合档案。

### 4.4 MassBattleLodSync 档案

| 维度 | 取值 |
| --- | --- |
| ClientPlaybackPolicy | 多数实体 `AuthoritativeInterpolation` 或 `HoldLatest` |
| InputPolicy | 玩家控制实体按玩法选择预测或服务器确认 |
| SnapshotPolicy | `BatchSnapshot` + `KeyFrameSnapshot` + `EventStream` |
| InterestPolicy | `DistanceAoi` + `TeamOrFactionAoi` + `PriorityBudget` + `LodFrequency` |
| RecoveryPolicy | `RequestAoiSlice` |
| ServerValidationPolicy | `AuthoritativeOnly` + `InputValidation` |
| TransportConditionPolicy | 必须接入带宽预算和弱网 profile |

这说明 `MassBattleLodSync` 的核心不是客户端 controller，而是服务端发布策略和兴趣管理。

### 4.5 FastReconnect 档案

| 维度 | 取值 |
| --- | --- |
| ClientPlaybackPolicy | 沿用断线前策略 |
| InputPolicy | 恢复输入上下文，处理断线期间输入缓存/丢弃 |
| SnapshotPolicy | `FullSnapshot` / `KeyFrameSnapshot` / `AoiSlice` |
| InterestPolicy | 沿用断线前 AOI |
| RecoveryPolicy | `ReconnectResume` |
| ServerValidationPolicy | 校验 session、world、battle、last frame/hash |
| TransportConditionPolicy | 与同步策略无关 |

这说明 `FastReconnect` 不应是同步模型，而是所有模型都可能使用的生命周期恢复能力。

## 5. Shooter / Moba 当前能力矩阵

| 能力维度 | Shooter 当前状态 | Moba 当前状态 | 缺口 |
| --- | --- | --- | --- |
| ClientPlaybackPolicy | `PredictRollback` 与 `AuthoritativeInterpolation` 均已存在 | `AuthoritativeInterpolation` 已接框架 playback | Moba 本地英雄预测尚未建模；Shooter 混合对象分层未显式化 |
| InputPolicy | 已有输入提交、accepted/current/status/server ticks | 当前偏服务器/播放侧，输入策略未进入同步框架 | 需要把 input acceptance/remap/reject 提升为通用诊断/策略 |
| SnapshotPolicy | full/packed/delta/keyframe/authority override 有协议基础 | 远端样本播放已验证 | delta import、keyframe 恢复、batch snapshot 未框架化 |
| InterestPolicy | 小规模默认全量 | 小规模默认全量 | AOI、LOD、优先级、带宽预算都缺 |
| RecoveryPolicy | 漂移检测、full snapshot 恢复、catch-up 有文档和部分实现 | 插值饥饿可诊断，但恢复策略弱 | full/keyframe/slice 请求应成为通用能力 |
| ServerValidationPolicy | server authoritative，lag compensation helper 已有 | server authoritative | 输入合法性、反作弊 envelope、hash audit 未框架化 |
| TransportConditionPolicy | NetworkConditionProfile 已可用于 Harness | 同上 | 还未和服务端发布预算/优先级联动 |
| Diagnostics | Reconciliation、InterpolationDiagnostics、DemoHarnessMetrics | Interpolation + Harness telemetry | 需要统一 sync event/health，而不是只靠 reconciliation report |

## 6. 抽象层修正建议

### 6.1 保留现有接口，但收窄命名语义

`IClientSyncStrategy<TInput, TSample>` 当前不要立刻删除或大改。它已经承接了 Shooter/Moba 两条线，短期应把文档语义修正为“客户端同步策略”，而不是“完整同步方案”。

建议后续逐步引入更小的能力接口：

```csharp
public interface IRemoteSnapshotConsumer<TSample>
{
    void ObserveRemote(in TSample sample);
}

public interface ILocalInputSubmitter<TInput>
{
    void SubmitInput(in TInput input);
}

public interface ISyncRecoveryRequester
{
    bool TryRequestRecovery(in SyncRecoveryRequest request);
}

public interface ISyncDiagnosticsProvider<TDiagnostics>
{
    TDiagnostics GetDiagnostics();
}
```

这些接口可以由现有 controller 选择性实现，避免把所有模式都塞进一个主接口。

### 6.2 把 `NetworkSyncModel` 降级为兼容档案名

短期保留枚举，避免破坏现有工厂和测试。但文档中应明确：

- `PredictRollback` / `AuthoritativeInterpolation` 是已落地的客户端能力档案。
- `HybridHeroPrediction` / `MassBattleLodSync` 是组合档案，不是底层模式。
- `FastReconnect` / `ServerRewindLagCompensation` 是附加 capability，不应长期作为互斥模型。

中期新增一个结构表达能力档案：

```csharp
public readonly struct NetworkSyncProfile
{
    public NetworkSyncModel CompatibilityModel { get; }
    public ClientPlaybackPolicy ClientPlayback { get; }
    public InputPolicy Input { get; }
    public SnapshotPolicy Snapshot { get; }
    public InterestPolicy Interest { get; }
    public RecoveryPolicy Recovery { get; }
    public ServerValidationPolicy ServerValidation { get; }
}
```

### 6.3 DemoHarness 改成验证能力矩阵，而不是假设全组合有效

当前 DemoHarness 的 A/B/C 三轴仍然有价值，但文档应把“任意组合都能跑”改成“任意声明支持的组合都能跑，缺能力可被显式报告”。

后续 `ISyncDemoCarrier` 可扩展能力声明：

```csharp
public interface ISyncDemoCarrierCapabilities
{
    string CarrierName { get; }
    bool Supports(in NetworkSyncProfile profile);
}
```

这样 Moba 不支持 PredictRollback、Shooter 不支持 MassBattle AOI 时，都可以成为矩阵里的显式 unsupported，而不是运行期误配。

### 6.4 建立统一时间锚点

后续应补框架层时间类型，避免 `CurrentFrame`、`TimelineTicks`、server ticks、input frame 混用：

```csharp
public readonly struct SyncFrame
{
    public int Value { get; }
}

public readonly struct SyncTimelineTicks
{
    public long Value { get; }
}

public readonly struct SyncTimeAnchor
{
    public string WorldId { get; }
    public SyncFrame StartFrame { get; }
    public long StartServerTicks { get; }
    public int ServerTickFrequency { get; }
    public float FixedDeltaSeconds { get; }
}
```

这会直接支撑 late join、reconnect、catch-up、interpolation playback、snapshot ordering。

### 6.5 诊断拆成事件和健康度

`SyncReconciliationReport` 应保留，但只负责预测回滚/校正类诊断。另补统一健康度事件：

- snapshot received / dropped / stale / gap
- interpolation starved / recovered
- rollback started / replay completed
- full snapshot requested / applied
- keyframe requested / applied
- AOI slice requested / applied
- input accepted / remapped / rejected
- lag compensated validation accepted / rejected

DemoHarness 后续汇总这些事件，比继续膨胀 `SyncReconciliationReport` 更稳。

## 7. 推荐迁移顺序

1. **文档先行**：把本审计结论同步回《同步演示框架示例层重构设计》和《网络同步框架阶段性设计》，修正“全组合自动成立”的表述。
2. **能力档案草案**：新增 `NetworkSyncProfile` 与各 policy enum 的设计文档，暂不改 runtime API；可执行草案见《网络同步能力档案与DemoHarness矩阵设计》。
3. **能力声明**：给 DemoHarness 增加 carrier capability 声明，让 unsupported 成为一等结果；capability 返回值、degraded 语义和 legacy fallback 见《网络同步能力档案与DemoHarness矩阵设计》。
4. **时间锚点**：抽 `SyncTimeAnchor` / frame / ticks 类型，优先服务 late join、reconnect、interpolation。
5. **诊断事件**：新增 sync event/health 结构，DemoHarness 先消费，不急着替换旧 report。
6. **枚举收敛**：在已有调用稳定后，再决定 `NetworkSyncModel` 是保留为 profile alias，还是拆成 profile registry。
7. **实现新能力**：再推进 BatchSnapshot、AOI/LOD、FastReconnect、LagCompensation 与 Hybrid 档案。

---

## 8. 判断标准

后续每新增一个同步能力，都用下面的问题审查：

1. 它是客户端播放策略、输入策略、快照发布策略、兴趣管理策略、恢复策略、服务器判定策略，还是传输条件？
2. 它是否与已有能力互斥？如果不是，不能放进单选模型里。
3. 它是否需要服务端和客户端同时实现？如果需要，不能只做 client controller 抽象。
4. 它是否依赖统一时间锚点？如果依赖，先补时间类型再写策略。
5. 它的失败和降级路径是什么？DemoHarness 必须能报告 unsupported / degraded / failed，而不是只报告 completed。

---

## 9. 总结

当前同步框架最值得保留的是“通用能力下沉框架、玩法只做强类型适配”的方向。最需要修正的是“同步模型”这个词承载了太多层级。下一阶段不要急着实现更多枚举项，而应该先把能力维度、支持矩阵、时间锚点和诊断事件稳定下来；其中 `NetworkSyncProfile`、policy enum、carrier capability 与 DemoHarness run status 的落地草案见《网络同步能力档案与DemoHarness矩阵设计》。

一句话：从“选择一种同步模式”升级为“声明一组同步能力档案”，AbilityKit 后续才能自然覆盖预测回滚、权威插值、低频批量、大规模 AOI、快速重连和延迟补偿这些并不互斥的真实组合。
